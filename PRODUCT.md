# Product

## Register

product

## Users

Salto serves readers who encounter unfamiliar words or short phrases while reading web pages in Chrome or Chromium-based browsers. They are already in the middle of a reading task and want quick help without losing context, opening another app, or turning reading into a separate study session.

The primary workflow is selection-based: choose text, inspect a nearby translation panel, save the word when useful, and return to the page. Later, the same saved vocabulary supports lightweight review and learning surfaces.

## Product Purpose

Salto is a browser extension for reading-time translation, word saving, and later vocabulary learning. Its purpose is to turn small moments of reading friction into reusable vocabulary memory while keeping the reading flow intact.

MVP success means users can translate selected text intentionally, save words immediately, see saved words enriched in the background, and trust that vocabulary and learning records remain local-first and sync-ready.

## Brand Personality

Salto should feel precise, lightweight, and focused.

The interface should behave like a quiet reading tool: present exactly the information needed, make state changes clear, and avoid ornamental UI that competes with the source page. It should feel technically reliable without becoming heavy or academic.

## Anti-references

Salto should not use gradient-led visual styling, decorative color washes, or AI-SaaS-style gradient cards.

It should also avoid excessively rounded components. Buttons, panels, settings controls, and learning cards should use restrained radii that feel native to a compact productivity tool rather than soft, toy-like, or overly playful.

Avoid designs that interrupt reading, over-gamify vocabulary learning, or turn simple translation and save actions into large modal workflows.

## Design Principles

1. Preserve the reading flow: every interaction should help the user understand or save text without pulling attention away from the page for longer than necessary.
2. Make state explicit: saved, loading, failed, enriched, ready, and retry states should be visible and unambiguous.
3. Prefer compact confidence: use familiar product UI patterns, restrained density, and predictable controls over decorative novelty.
4. Separate transient help from durable memory: translation panel output and saved vocabulary enrichment are different jobs, and the UI should make that distinction understandable.
5. Keep future sync in view: local-first MVP surfaces should not imply account, cloud, or cross-device features that are intentionally deferred.

## Accessibility & Inclusion

Target WCAG AA for contrast, focus visibility, and keyboard access. The extension should support reduced motion, avoid color as the only status indicator, and keep floating UI usable near viewport edges and on dense web pages.

Because Salto appears on top of arbitrary websites, foreground panels and triggers need strong contrast, clear focus states, predictable dismissal with `Esc`, and careful behavior around page text selection.
